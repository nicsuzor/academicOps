# aops v0.3 Release Acceptance Tests

This file defines acceptance tests for the aops v0.3 release. The focus is on hydrator routing correctness: does the hydrator recommend the right workflow for realistic academic prompts, especially as the workflow catalog grows?

## Prerequisite: Context Loading

The v1.1 tests failed because the hydrator subagent was invoked without session context (SKILLS.md, WORKFLOWS.md, etc. were not preloaded). Before running these tests, verify that the hydrator receives its full input file containing:

- Skills Index (from SKILLS.md)
- Workflows Index (from WORKFLOWS.md)
- Project-scoped workflows (from `.agent/workflows/*.md`)

If the hydrator output says "out of scope" or fails to reference any workflows/skills, the test harness is broken, not the routing logic. Log the harness gap and halt.
Validates that the hydrator correctly routes prompts to the right skill or workflow for 12 of the indexed skills.

## Instructions for Test Runner Agent

You are the test runner. For each test:

1. **Build the context file** — run:
   ```
   uv run scripts/build_hydrator_test_input.py "<user input>"
   ```
   This prints a file path containing the full SKILLS.md/WORKFLOWS.md context.

2. **Invoke the hydrator** with that path:
   ```
   Agent(subagent_type='aops-core:hydrator', prompt='<path from step 1>')
   ```

3. **Evaluate pass criteria** — read the hydrator output and judge semantically.

4. **Update the Results table** — mark PASS/FAIL with timestamp and notes.

Do NOT invoke the hydrator with the raw user prompt — it will receive no framework
context and all routing tests will fail. Always build the context file first.

When all tests complete, summarize: X passed, Y failed, Z errors.

### TEST-001: Decompose vs Feature-Dev Discrimination (Goal-Level)

**ID**: `v0.3-decompose-goal-level`

**Description**: When a user describes a multi-month, uncertain-path goal, the hydrator must route to the decompose workflow, not feature-dev. This is the hardest routing distinction in the decision tree.

**User Input**:

```
I need to write a paper on platform governance for a special issue. The deadline is in 3 months. I have some preliminary data from last year's survey but haven't done the analysis yet. Where do I start?
```

**Expected Behavior**:

1. Hydrator identifies this as a multi-session, uncertain-path goal
2. Hydrator recommends the `decompose` workflow
3. Hydrator identifies scope as multi-session
4. Hydrator does NOT recommend feature-dev (this is not a "known what to build" situation)

**Invocation Method**: `hydrator-only`

**Pass Criteria** (ALL must be true):

1. Hydrator output references `[[decompose]]` or `decompose` workflow
2. Hydrator output classifies scope as multi-session (or equivalent: "goal-level", "multi-month")
3. Hydrator output does NOT recommend `[[feature-dev]]` as the primary workflow
4. Hydrator output does NOT recommend `[[simple-question]]`

**Why This Matters**: The decompose-vs-feature-dev boundary is the most common misroute. Feature-dev is for "I know what to build, design how"; decompose is for "I need to figure out what to do." An academic planning a paper is squarely in decompose territory. If the hydrator routes this to feature-dev, the agent will skip the critical planning phase and jump into execution prematurely.

**Related**:

- Workflow: `[[workflows/decompose]]`
- Decision tree: "Goal-level / multi-month work?" branch

## Test Design Rationale

### Coverage Matrix

| Failure Mode                    | Tests That Cover It                    |
| ------------------------------- | -------------------------------------- |
| Decompose/feature-dev confusion | TEST-001, TEST-008                     |
| Academic workflow misrouting    | TEST-002, TEST-003, TEST-004           |
| Over-routing (false escalation) | TEST-005, TEST-006                     |
| Project-scoped workflow loading | TEST-007                               |
| Multi-intent handling           | TEST-009                               |
| Skill awareness in routing      | TEST-003, TEST-006, TEST-010, TEST-011 |
| Batch vs decompose confusion    | TEST-012                               |
| v1.1 regression coverage        | TEST-010, TEST-011                     |

### Hard Routing Cases (by design)

- **TEST-001**: Paper writing could be feature-dev ("I know I need a paper") or decompose ("I need to figure out the path"). The "where do I start?" signals uncertainty.
- **TEST-004**: Report-finalization and review-response are sibling workflows with overlapping vocabulary ("comments", "revision", "feedback"). The discriminator is whether the user is revising content or annotating what changed.
- **TEST-005**: Contains academic keywords ("ARC", "grant") that could trigger peer-review via naive keyword matching, but the intent is purely informational.
- **TEST-009**: Two intents in one prompt -- the hydrator must not collapse them.
- **TEST-012**: "Review 15 reports" sounds like decomposition but is actually batch processing of a known workflow.
